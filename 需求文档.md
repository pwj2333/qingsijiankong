# 勤思船舶平台自动处理异常报警系统需求文档

## 1. 项目背景

为减少人工重复登录平台、手动查询异常报警、逐条处理告警的工作量，建设一套运行在 Windows Server 上的自动化处理系统。

系统通过调用勤思船舶系统提供的接口，定时查询异常报警记录，并按照预设规则自动完成告警处理，同时记录处理日志，便于后续审计、排查和人工介入。

## 2. 建设目标

本系统需要实现以下目标：

1. 每隔 30 秒自动查询一次异常报警。
2. 自动识别“未处理”的异常告警。
3. 按照预先配置的规则自动调用“处理异常告警”接口。
4. 记录每次查询、处理结果、失败原因和运行状态。
5. 可部署在 Windows Server 上长期稳定运行。
6. 支持后续扩展更复杂的处理规则。

## 3. 需求范围

本期建设范围包括：

1. 调用登录接口获取 `accessToken`。
2. 自动维护 `accessToken`，过期后重新获取。
3. 定时轮询异常报警接口。
4. 过滤出未处理告警。
5. 根据规则自动生成处理参数并提交处理接口。
6. 对已处理过的告警进行去重，避免重复提交。
7. 输出本地日志文件。
8. 通过配置文件维护账号、密码、轮询周期、处理规则等参数。
9. 提供 Windows Server 下的启动方式。

本期暂不包含：

1. 图形化管理后台。
2. 短信、微信、邮件通知。
3. 数据库存储。
4. 多平台多租户管理。
5. 人工审批流。

## 4. 接口依据

根据现有接口文档，本系统依赖以下 3 个接口：

1. 获取授权 `accessToken`
   地址：`https://haiwang.kinthtech.com.cn/msaPlatform/platform/user/login`
2. 查询异常报警
   地址：`https://haiwang.kinthtech.com.cn/ferryEye/captain/findExpAlarm`
3. 处理异常告警
   地址：`https://haiwang.kinthtech.com.cn/ferryEye/captain/expAlarmHanding`

接口文档说明中，登录密码需要先做 MD5 后再提交。

## 5. 用户角色

本系统本期主要面向 1 类使用者：

1. 系统运维人员
   负责部署程序、配置账号密码、查看日志、启动和停止服务。

后续如有需要，可增加业务管理员角色，用于维护处理规则。

## 6. 业务流程

### 6.1 主流程

1. 系统启动。
2. 读取本地配置文件。
3. 调用登录接口获取 `accessToken`。
4. 每隔 30 秒调用一次“查询异常报警”接口。
5. 获取状态为“未处理”的报警记录。
6. 根据预设规则判断该告警如何处理。
7. 调用“处理异常告警”接口提交处理结果。
8. 写入本地日志。
9. 进入下一轮轮询。

### 6.2 Token 维护流程

1. 系统启动时登录一次。
2. 当接口返回鉴权失败、Token 失效或接近过期时，重新登录获取新 Token。
3. 新 Token 自动替换旧 Token，后续请求统一使用新 Token。

### 6.3 告警处理流程

1. 查询到未处理告警。
2. 判断该告警是否已被本系统处理过。
3. 若未处理过，则根据规则生成处理参数：
   `id`、`isValid`、`handleMethod`、`handleResult`、`isArchive`、`remark`
4. 调用处理接口。
5. 若成功，记录成功日志并加入本地去重记录。
6. 若失败，记录失败日志，保留后续再次重试能力。

## 7. 功能需求

### 7.1 配置管理

系统需支持通过配置文件维护以下参数：

1. 登录账号。
2. 登录密码。
3. 轮询间隔，默认 30 秒。
4. 查询页码和分页大小。
5. 查询条件，如状态、开始时间、结束时间。
6. 自动处理默认参数。
7. 日志目录。
8. 去重记录文件位置。
9. 请求超时时间。
10. 是否开启调试日志。

### 7.2 登录鉴权

系统需支持：

1. 按接口要求将密码 MD5 加密后提交。
2. 从响应中提取 `accessToken` 和 `expiresIn`。
3. 将 `accessToken` 自动写入后续接口请求头 `Authorization`。
4. 登录失败时自动记录错误信息。

### 7.3 异常报警查询

系统需支持：

1. 每 30 秒自动执行一次查询。
2. 默认查询“未处理”告警，即文档中的 `status=2`。
3. 支持分页查询，避免单页数据过多导致漏单。
4. 支持按时间范围查询，默认查询最近 1 天数据。
5. 支持根据需要调整排序参数 `orderBy`。

### 7.4 自动处理规则

系统需支持按配置规则自动处理告警。

本期默认规则建议如下：

1. 对查询结果中的未处理告警自动执行处理。
2. 默认提交为“有效告警”或“无效告警”需通过配置指定。
3. `handleMethod` 支持配置固定值，如：
   `系统测试`、`识别错误`、`未正确匹配`
4. `handleResult` 支持配置固定模板。
5. `remark` 支持配置固定说明，注明由自动化系统处理。
6. `isArchive` 默认传 `0`。

说明：
由于当前接口文档仅说明了处理接口字段，并未给出明确的“自动判断有效/无效”的业务规则，因此本期默认采用“固定规则自动处理”的方式，后续再扩展为按船名、行为类型、风险等级等条件匹配不同策略。

### 7.5 去重控制

系统需支持：

1. 记录已成功处理的告警 ID。
2. 避免同一条告警被重复调用处理接口。
3. 程序重启后仍可识别历史已处理记录。

### 7.6 日志管理

系统需记录以下日志：

1. 程序启动日志。
2. 登录成功和失败日志。
3. 每次查询的开始与结束日志。
4. 查询到的告警数量。
5. 每条告警的处理结果。
6. 接口异常、超时、返回失败日志。
7. 重试和重新登录日志。

### 7.7 Windows Server 运行

系统需满足以下部署要求：

1. 可在 Windows Server 环境下运行。
2. 支持命令行直接启动。
3. 支持通过 `bat` 启动脚本运行。
4. 支持后续扩展为 Windows 计划任务或 Windows 服务方式常驻运行。

## 8. 非功能需求

### 8.1 稳定性

1. 单次接口失败不应导致程序退出。
2. 程序异常时应输出清晰错误日志。
3. 网络抖动或接口超时后，应能在下一轮继续运行。

### 8.2 可维护性

1. 代码结构清晰，便于后续增加新规则。
2. 配置与业务逻辑分离。
3. 日志便于人工排查问题。

### 8.3 安全性

1. 账号密码不得硬编码在主程序中。
2. 凭据应放在配置文件中管理。
3. 日志中不得输出明文密码。

### 8.4 性能

1. 正常情况下单轮查询和处理应在 30 秒轮询周期内完成。
2. 支持常规规模告警列表处理，不要求高并发。

## 9. 建议技术方案

结合部署环境和需求复杂度，本期建议采用 Python 开发，原因如下：

1. 适合快速对接 HTTP 接口。
2. 在 Windows Server 上部署简单。
3. 易于通过配置文件和日志文件进行运维。
4. 后续扩展为服务程序成本较低。

建议系统结构如下：

1. `config`：配置读取
2. `client`：接口请求封装
3. `auth`：登录与 Token 管理
4. `poller`：定时轮询
5. `handler`：告警处理规则
6. `store`：本地去重记录
7. `logger`：日志输出
8. `main`：程序入口

## 10. 输入输出

### 10.1 输入

1. 勤思平台账号
2. 勤思平台密码
3. 轮询配置
4. 自动处理规则配置

### 10.2 输出

1. 已处理告警结果
2. 运行日志文件
3. 已处理告警 ID 记录文件

## 11. 异常处理要求

系统需考虑以下异常情况：

1. 登录失败
2. Token 失效
3. 接口超时
4. 接口返回非成功码
5. 查询结果字段缺失
6. 处理接口调用失败
7. 本地日志文件写入失败

异常发生时要求：

1. 日志中记录详细原因。
2. 程序尽量不中断。
3. 下一轮继续自动运行。

## 12. 本期默认处理策略

在未补充更细业务规则前，本期建议按以下默认策略实施：

1. 每 30 秒查询一次 `status=2` 的未处理异常报警。
2. 对查询到且未处理过的告警，自动调用处理接口。
3. 处理参数默认由配置指定，例如：
   `isValid=0`
   `handleMethod=识别错误`
   `handleResult=` 空
   `isArchive=0`
   `remark=` 空
4. 同一告警 ID 不重复处理。

说明：
如果你的真实业务规则不是“全部自动处理”，而是“只处理某些船舶、某些异常行为、某些风险等级”，则需要在编码前补充规则清单。

## 13. 待确认事项

以下内容目前接口文档中没有完全明确，建议在编码前由你确认：

1. 是否所有未处理告警都需要自动处理？
2. 自动处理时，`isValid` 默认应传 `0` 还是 `1`？
3. `handleMethod` 默认要使用哪个固定值？
4. `handleResult` 和 `remark` 需要写成什么内容？
5. 是否需要按船名、行为类型、风险等级做差异化处理？
6. 是否需要处理成功后自动归档，即 `isArchive` 是否要改为 `1`？
7. 是否只处理最近 1 天数据，还是要处理更长时间窗口？
8. 程序最终希望以哪种方式运行：
   命令行常驻、计划任务、还是 Windows 服务？

## 15. 验收标准

满足以下条件即可视为本期验收通过：

1. 程序可在 Windows Server 上成功启动。
2. 程序能够成功登录并获取 `accessToken`。
3. 程序每 30 秒自动查询一次异常报警。
4. 程序可识别未处理告警并自动调用处理接口。
5. 同一条告警不会被重复处理。
6. 异常情况下程序不会轻易退出。
7. 能输出可追踪的运行日志。
